# 老师提出的主线需求

+ 首页：默认树图除了展示 TE，还可以展示 ICD-11 的树图
+ 问答助手
  + 要可以对图谱进行反作用，让左侧同步右侧的回答，同时显示文献
  + 要具备分析拓扑、图的结构啥的相关的能力，比如问“哪些疾病与 TE 关联度最高”
+ 点击箭头要显示信息，类似于 Cyt，展示如
```text
LINE1 -> participates in -> Hepatocellular carcinoma
Evidence: PMID: 11007163, 23540693, 23665280, 27242996, 27293999, 29643204, 30717368, 32980960, 40421600
```
之类的数据
+ 点击疾病分类节点要可以跳转，也可以接着继续跳转
+ 搜索：加入 TE 的更多信息，比如测序等

# 实施主线拆解

## 1. 首页支持 TE 树 / ICD-11 树切换

### 大致思路
- 现在首页默认展示的是 TE 分类树，后续要扩展成“树图入口层”。
- 页面上至少支持两种树：
  - TE 树
  - ICD-11 树
- 这件事本质上是做“默认树入口”的扩展，不是做动态图本身。
- 更稳的实现方式是：
  - 保留现有 TE 树渲染链路
  - 为 ICD-11 准备一套相同结构的树数据
  - 在首页或 preview 中加入树类型切换

### 前置要求
- 先确认 ICD-11 数据范围：
  - 是完整 ICD-11 层级
  - 还是只保留和当前疾病相关的部分
- ICD-11 数据要整理成和 TE 树类似的层级格式，至少包含：
  - `id`
  - `label`
  - `children`
- 当前阶段优先考虑英文版数据即可，不因为中英文双支持阻塞开发

### 难度
- `中`

### 风险点
- ICD-11 层级如果太大，首页默认树可能过于拥挤。
- ICD-11 节点命名如果和当前数据库疾病名不一致，后续点击跳转会不稳定。

## 2. 问答助手升级为“图谱问答 + 文献支撑 + 图谱联动”

### 大致思路
- 当前 QA 已经恢复到可用状态，但主要还是“围绕当前图回答”。
- 老师要的是下一步：
  - 回答能反作用左侧图谱
  - 回答里能带文献
  - 回答还能解释图结构和拓扑
- 这条主线可以拆成三层：
  1. 问答能读取当前图上下文
  2. 问答能返回结构化结果
  3. 回答可以驱动图谱变化

### 前置要求
- 需要先明确 QA 的输入上下文格式：
  - 当前中心节点
  - 当前 key-node level
  - 当前图中的 nodes / edges
  - 当前 fixed-view 状态
- 需要确认文献展示方式：
  - 直接在回答中列 PMID
  - 还是额外生成证据卡片
- 如果要“反作用图谱”，需要先定义动作类型：
  - 高亮已有节点
  - 跳到新 query
  - 切换为回答对应的局部子图

### 难度
- `高`

### 风险点
- 这是最复杂的一条，因为它同时涉及：
  - QA 提示词
  - 图谱数据摘要
  - 图谱动作映射
  - 文献证据格式
- 如果一开始做得太重，容易出现 QA 能说但不能稳定驱动图。

### 关于“structure / topology”我们到底要做什么

#### structure
- 更偏向描述“这张图是怎么组成的”，例如：
  - 当前图有哪些类型节点
  - 哪一类节点最多
  - 当前图是单中心、树状还是多团簇
  - 哪些疾病分类在当前图中最突出
  - 当前中心节点主要连向哪些类别

#### topology
- 更偏向描述“这张图的连接关系和节点角色”，例如：
  - 哪些节点 degree 最高
  - 哪些疾病与某个 TE 关联度最高
  - 哪些节点像 hub
  - 哪些节点是桥梁节点
  - 两个节点之间最短路径是什么

### 第一版建议先做的能力
- 节点类型统计
- 高 degree 节点排行
- 当前中心节点的邻居排序
- 基于边数 / 证据数的“关联度最高”排序

### 后续再做的能力
- 最短路径
- 桥梁节点分析
- 团簇 / 社区分析
- 更正式的 centrality 指标

### 如果要实现“QA 反向驱动动态图”，建议按这几步改

当前先采用最稳的范围约束：
- 只驱动动态图
- 不驱动默认树
- 统一采用预设状态：
  - `key-node level = 1`
  - `fixed view = on`
  - `renderer` 保持当前页面已有模式
- QA 只负责返回一个“结构化子图结果”，前端按固定规则渲染

#### 第 1 步：先固定 QA 的输出格式
- 目标是让 QA 不返回自由文本动作，而是返回稳定结构，例如：
  - `query`
  - `nodes`
  - `edges`
  - `evidence`
- 第一版不要求 QA 决定页面状态，只要求它明确：
  - 当前问题围绕哪个实体
  - 用了哪些边
  - 左侧应该展示哪些节点和边
- 主要涉及：
  - `api/qa.php`
  - 本地 QA 模板 / prompt 资源

#### 第 2 步：前端让左侧动态图支持“回答后同步”
- 左侧始终是同一个 G6 动态图区，不再引入新的用户感知模式
- 当 QA 回答完成后：
  - 左侧直接同步为 AI 实际使用到的边和这些边两端的节点
- 这一步不要改默认树逻辑，只在现有动态图容器里切换数据内容
- 主要涉及：
  - `assets/js/renderers/g6/index-g6-shared.js`
  - `assets/js/renderers/g6/index-g6-runtime.js`
  - `assets/js/renderers/g6/index-g6-qa.js`

#### 第 3 步：把 QA 回答中的结构化边送进左侧图
- 当 QA 返回结构化结果后：
  - 左侧不再重新猜测 query
  - 直接按 QA 给出的 nodes / edges 同步当前动态图
- 这一版不做复杂布局分支，统一沿用当前动态图布局参数
- 主要涉及：
  - `assets/js/renderers/g6/index-g6-qa.js`
  - `assets/js/renderers/g6/index-g6-shared.js`

#### 第 4 步：加“回到普通图”的退路
- QA 驱动图谱后，用户需要能一键回到普通 query 图
- 这一步很重要，否则用户容易被困在当前回答对应的图里
- 第一版建议提供：
  - `Back to graph`
  - 或 `Restore query graph`
- 主要涉及：
  - `assets/js/renderers/g6/index-g6.bootstrap.js`
  - `assets/js/renderers/g6/index-g6-qa.js`

#### 第 5 步：最后再补文献联动
- 当回答后同步图稳定后，再把回答里引用的 PMID 和左侧边证据联动起来
- 这一层可以晚一点做，不要和“先把子图驱动起来”绑在一起
- 主要涉及：
  - `api/qa.php`
  - `assets/js/renderers/g6/index-g6-qa.js`
  - `assets/js/renderers/g6/index-g6-shared.js`

### 这条功能为什么现在可以做
- 范围已经明确收紧：
  - 只驱动动态图
  - 不改树
  - 状态固定
- 这样它就不再是“任意控制整页”，而是“把 QA 识别到的边和节点渲染成一张固定规则的局部图”
- 在这个范围内，它和 Cyt 的实现思路已经很接近

### 第一版完成标准
- 用户提问，例如：
  - “与 LINE1 相关的疾病”
- QA 能返回结构化结果
- 左侧 G6 动态图切换成：
  - 中心对象为 `LINE1`
  - 只展示 QA 使用到的边和两端节点
  - `key-node level = 1`
  - `fixed view = on`
- 用户可以一键回到普通 query 图

## 3. 点击边显示关系和证据

### 大致思路
- 目标是让 G6 也具备和 Cyt 类似的边信息展示能力。
- 用户点击边后，要能看到：
  - `source`
  - `relation`
  - `target`
  - `evidence / PMID`
- 这件事属于“图谱解释能力”的基础功能，优先级很高，投入产出比也高。

### 前置要求
- 后端 `api/graph.php` 返回的 edge 数据里要稳定带有：
  - `relation`
  - `relationType`
  - `pmids`
  - `evidence`
- 前端需要统一边点击后的展示位置：
  - detail 区
  - 侧边说明区
  - 或浮层
- 要确认不同入口是否共用同一套展示：
  - preview
  - search
  - 独立 G6 页面

### 难度
- `低-中`

### 风险点
- 主要风险不是实现难，而是格式：
  - PMID 很长时怎么换行
  - 证据展示如何保持清晰
  - 不同页面是否一致

## 4. 疾病分类节点支持跳转与继续跳转

### 大致思路
- 现在疾病分类节点更多是视觉锚点。
- 老师的意思是：它不能只是装饰，而要成为真正可交互的导航节点。
- 点击疾病分类节点后，至少要支持两种行为：
  - 进入该分类的局部动态图
  - 在该分类内部继续下钻到具体疾病

### 前置要求
- 需要先定义点击疾病分类节点后的 query 语义：
  - 是按 `disease_class` 聚合检索
  - 还是生成该类疾病的专题图
- 后端可能需要支持新的查询方式，例如：
  - `type=disease_class`
  - `class=Neoplasms`
- 需要确认继续跳转的层级设计：
  - 分类 -> 疾病
  - 分类 -> 子图 -> 疾病

### 难度
- `中`

### 风险点
- 当前 `DiseaseClass` 是前端合成节点，不是数据库真实节点。
- 所以后续如果要让它跳转，后端要能理解分类语义，而不是只认普通实体名。

## 5. 搜索页补充更多 TE 信息

### 大致思路
- 现在搜索页已经有：
  - 最佳命中
  - Repbase 参考区
  - 局部图谱
- 老师希望的是：
  - 再补更多 TE 信息
  - 比如测序、序列摘要、注释等
- 这条主线更偏“信息增强”，不是图结构问题。

### 前置要求
- 需要先确认 TE 扩展信息来源：
  - 现有 Repbase 映射
  - 本地整理后的 TE 数据文件
  - 其他测序 / 注释来源
- 需要明确页面上想补充哪些字段，避免先做太散：
  - 标准名
  - 别名
  - 物种
  - 序列摘要
  - 测序 / 注释信息
  - 参考文献数

### 难度
- `中`

### 风险点
- 主要风险在数据来源是否稳定、字段是否统一。
- 不同 TE 的信息完整度差异可能很大，页面要能接受部分字段为空。
